# FunPairDL 專案詳細介紹

## 1. 文件用途

這份文件是給兩種讀者準備的：

1. 人類使用者或維護者，想快速理解這個專案到底在做什麼、怎麼跑、哪些模組負責哪些工作。
2. LLM 或自動化分析工具，想在最短時間內建立正確的系統心智模型，避免只看檔名就做出錯誤推論。

這份文件不是操作手冊而已，也不是單純的檔案清單。它會說明：

- 專案定位
- 架構分層
- 核心資料模型
- 實際下載流程
- Provider 行為差異
- GUI、嵌入式瀏覽器、瀏覽器擴充套件之間的關係
- API 介面
- 設定與持久化檔案
- 命名整理規則
- 目前測試覆蓋與已知風險

---

## 2. 專案一句話總結

`FunPairDL` 是一個偏向 EroScripts 生態使用的本機下載管理器，核心目標是把「影片連結」與「funscript 連結」整理成一個下載單位，再由本機後端負責解析來源、下載檔案、追蹤進度、保存佇列、以及在下載完成後把輸出檔案整理成便於播放器或裝置使用的命名結構。

它不是純 CLI 工具，也不是純瀏覽器擴充套件，而是由下列幾個部分組成：

- Windows 桌面 GUI
- 本機 FastAPI 服務
- 下載佇列與背景下載引擎
- 可嵌入 EroScripts 的內建瀏覽器
- Chrome/Chromium 擴充套件
- 多個 provider 解析器

---

## 3. 這個專案真正解決的問題

此專案處理的不是單純「下載一個 URL」，而是以下這種比較麻煩的實際工作流：

1. 在 EroScripts 或相關網站上找到一篇貼文。
2. 從貼文裡萃取影片來源與 funscript 連結。
3. 有些影片是 Pixeldrain、MEGA、GoFile，有些是 Iwara 或其他 yt-dlp 支援站點。
4. 有些連結是短網址、需要 cookies、需要登入狀態、需要 session。
5. 有些是一個 bundle，裡面可能有多個影片或多個 script。
6. 下載完成後，影片與 script 還要重新命名，甚至分成 `.alt`、`.alt1` 之類的變體資料夾，讓後續播放器或裝置能辨識。

因此這個專案本質上是：

- 一個「配對型下載任務管理器」
- 一個「多來源 URL 解析器」
- 一個「本機整合式下載工作站」

## 4. 專案的核心概念

這個專案最重要的概念不是單一檔案，而是 `Pair`。

### 4.1 Pair

`Pair` 代表一組下載任務，通常可視為：

- 一篇貼文
- 一個作品
- 一次送進來的影片 + script 集合

一個 `Pair` 下面可以有多個 `PairItem`。

### 4.2 PairItem

`PairItem` 代表一個實際檔案項目，例如：

- 一支影片
- 一個 `.funscript`
- bundle 展開後的某個檔案

### 4.3 Bundle

某些連結不是單一檔案，而是一個集合，例如：

- Pixeldrain list
- MEGA folder

這些 URL 先以「bundle placeholder」加入佇列，之後再展開成多個 `PairItem`。

### 4.4 Provider

不同站點的 URL 解析方式不同，因此專案把來源站點抽象成 provider，例如：

- `pixeldrain`
- `mega`
- `gofile`
- `eroscripts`
- `iwara`
- `ytdlp`
- `direct`

### 4.5 Organize Output

下載完成後，不是單純保留原始檔名，而是會依規則：

- 統一影片檔名
- 統一 script 檔名
- 依軸向 suffix 保留多軸資訊
- 遇到變體或多作者時建立 `.alt`、`.alt1` 等子資料夾
- 用 hardlink 複製影片到變體資料夾

## 5. 架構總覽

可把整體架構理解成下面這條鏈：

```text
使用者 / EroScripts 頁面 / 瀏覽器擴充套件
        ↓
GUI 或 Extension / 內嵌 Browser
        ↓
本機 API (`/api/*`)
        ↓
QueueManager
        ↓
ProviderRegistry
        ↓
DownloadTask / MEGA 專用流程 / yt-dlp HLS 流程
        ↓
queue.json / config.json / 實際下載檔案 / log
```

更具體一點：

- GUI 負責操作與展示。
- API 是 GUI、嵌入瀏覽器、外部擴充套件的統一入口。
- `QueueManager` 是整個專案的中樞。
- Provider 負責把「原始 URL」轉成「可下載檔案」。
- 下載層再依檔案特性選擇分段下載、HLS 下載或 MEGA 特殊下載。

## 6. 執行模式

### 6.1 GUI 模式

入口：

- `run.py`
- `FunPairDL.pyw`
- `funpairdl/app.py`

GUI 模式會同時啟動：

- PySide6 視窗
- 內建分頁瀏覽器
- qasync 整合的 asyncio 事件迴圈
- 本機 FastAPI 服務
- QueueManager

適合一般使用者直接開啟視窗操作。

### 6.2 Server 模式

入口：

- `run_server.py`
- `funpairdl/server.py`

Server 模式不啟動 GUI，只啟動：

- QueueManager
- API server
- 自動儲存

這個模式是給瀏覽器擴充套件或其他外部工具使用的輕量後端。

### 6.3 Extension / 內建瀏覽器模式

這個專案同時存在兩種前端整合：

1. 真正的瀏覽器擴充套件
2. GUI 內的 QWebEngine 嵌入式瀏覽器

這兩者不是彼此無關，而是刻意共用核心邏輯：

- `extension/content.js`
- `extension/content.css`

GUI 內建瀏覽器會把這些腳本注入頁面，所以內建瀏覽器與瀏覽器擴充套件的「頁面分析與送出體驗」基本上是同一套邏輯。

### 6.4 主要技術棧與依賴

從 `requirements.txt` 可直接看出的核心依賴有：

- `PySide6`：桌面 GUI、QWebEngine、QWebChannel
- `aiohttp`：HTTP client
- `yt-dlp`：多站點影片解析與部分串流下載
- `mega.py`：MEGA 相關能力，雖然主下載路徑已大量轉到自訂 `mega_api`
- `fastapi`：本機 API
- `uvicorn`：API server 執行
- `pydantic`：API schema
- `qasync`：Qt 與 asyncio 整合

另外還有幾個「程式碼會用到，但不一定在 `requirements.txt` 裡」的輔助依賴：

- `browser_cookie3`：EroScriptsProvider 嘗試從本機瀏覽器抽 cookie 時會用到
- `Pillow`：icon 生成腳本會用到
- `winshell` / `pywin32`：建立 Windows 捷徑時會動態安裝

這意味著：

- runtime 主體依賴是明確的
- 部分便利工具或選用功能有隱含依賴
- 如果未來要整理封裝、CI、或最小安裝包，這些隱含依賴值得獨立盤點

`Pair` 狀態：

- `queued`
- `downloading`
- `paused`
- `completed`
- `failed`

### 8.2 `ItemState`

`PairItem` 狀態：

- `pending`
- `resolving`
- `downloading`
- `paused`
- `completed`
- `failed`

### 8.3 `FileType`

- `video`
- `funscript`
- `other`

### 8.4 `PairItem` 的重要欄位

- `url`：原始 URL
- `resolved_url`：解析後 URL
- `filename`：當前檔名，可能會被 provider 或 organize 流程更新

- `organized`：是否已完成 organize
- `original_filenames`：供 undo 用的原始檔名對照

## 9. 完整下載流程

這是專案最核心的段落。

### 9.1 任務建立

任務可從三個入口進來：

1. GUI 手動新增
2. 瀏覽器擴充套件送到 `/api/pair`
3. 擴充套件右鍵單連結送到 `/api/link`

`QueueManager.add_pair()` 會：

- 建立 `Pair`
- 根據 URL 推測 provider
- 對影片與 script 建立 `PairItem`
- 對 bundle URL 先放 placeholder
- 對 script 保留作者資訊
- 避免相同輸出目錄的未完成任務重複加入

### 9.2 Queue Pump

`QueueManager` 有一個常駐的 pump 迴圈：

- 依設定的 `max_concurrent_pairs` 同時啟動多個 pair
- 每個 pair 各自跑自己的下載流程
- pump 具有 heartbeat 與 watchdog
- 若 pump 卡死超過一定時間，會自動重啟
- GUI/API 也能手動觸發 `force_restart_pump()`

### 9.3 Bundle 展開

若 `Pair` 裡有 bundle 項目，會先做 `_resolve_bundles()`：

- Pixeldrain list 會展開成多個 `ResolvedFile`
- MEGA folder 會先 probe，再轉成多個 `PairItem`

展開後，如果變成「一個 bundle 內有多支影片 + 多組 script」，就可能再觸發 `_auto_split_bundle_pair()`：

- 依檔名 stem 嘗試把 script 對應到影片
- 自動拆成多個 `Pair`
- 原始 bundle pair 會變成空的 completed pair

這是一個非常重要的設計點：  
這個系統不只會「下載」，也會「根據 bundle 內容重建任務結構」。

### 9.4 URL 解析

每個 item 會透過 `ProviderRegistry` 找到對應 provider，再得到 `ResolvedFile`。

解析後可能得到：

- 直接 HTTP URL
- HLS / DASH 類型串流
- MEGA 特殊下載標記
- 額外 headers
- 檔名
- 檔案大小

### 9.5 實際下載分流

下載路徑分三類：

1. 一般 HTTP 檔案  
   使用 `DownloadTask` + `SegmentDownloader`

2. MEGA 檔案  
   由 `QueueManager._download_mega()` 走 `funpairdl.utils.mega_api`

3. HLS / DASH / 需 yt-dlp 實際下載的來源  
   由 `QueueManager._download_hls()` 走 yt-dlp

### 9.6 一般 HTTP 檔案下載

`DownloadTask` 會：

- 先做 HEAD 解析
- 判斷是否支援 range
- 決定分段數量
- 建立 `.parts` 暫存檔
- 啟動多個 `SegmentDownloader`
- 做 round-based segment retry
- 最後 merge segment 檔案

這個流程有幾個關鍵特性：

- 小檔案或不支援 range 的來源會自動退化成單段下載
- segment retry 是「只重試失敗 segment」，不是整檔重來
- merge 完成後會清理暫存 segment

### 9.7 MEGA 下載

MEGA 不是一般 HTTP 下載，原因是它有加密與 session 邏輯。

實際上：

- `providers/mega_provider.py` 主要扮演「辨識這是 MEGA」的角色
- 真正的 probe / SID 驗證 / 下載 / 解密邏輯在 `utils/mega_api.py`

GUI 內建瀏覽器還會嘗試自動取得或刷新 `mega_sid`：

- 先背景開啟 `mega.nz`
- 若已登入就讀取 `u_sid`
- 若未登入且設定內有帳密，就嘗試自動登入
- 成功後把 SID 存回 `config.json`

### 9.8 HLS / yt-dlp 下載

對 Iwara 或一般 yt-dlp 類站點：

- provider 先用 yt-dlp 抽 metadata
- 若最終 URL 是 `.m3u8` 或來源站需要 yt-dlp 下載，`ResolvedFile.is_hls=True`
- 之後 `QueueManager` 走 `_download_hls()`
- 畫質會參考 `preferred_resolution`

### 9.9 Item 級重試

在 segment retry 之外，pair download 還有 item 級重試：

- 若 item 失敗，系統會重新 resolve URL
- 避免 CDN 簽名過期或暫時性來源失效
- 預設有額外重試輪次與退避延遲

### 9.10 Crash Recovery / Resume

專案有幾層恢復能力：

- 啟動時把 `queue.json` 裡的 `DOWNLOADING` pair 轉回 `QUEUED`
- 若 final output 已存在且大小正確，直接標記完成
- 若 segment 檔其實都已下載完，會嘗試 merge 回正式檔案
- pause/cancel 後的 partial segment 會保留，供下次續跑

## 10. 下載層的技術重點

### 10.1 QueueManager 有獨立下載執行緒

這不是單純在 Qt 主執行緒裡跑 asyncio。

`QueueManager.start()` 會建立：

- 一個獨立 thread
- 該 thread 自己的 asyncio event loop
- 該 loop 內的 `aiohttp.ClientSession`
- pump task

目的很明確：

- 下載 I/O 不阻塞 GUI
- Qt 只負責畫面與 signal/slot
- 真正重的網路與下載工作放到背景 thread

### 10.2 SegmentDownloader 的磁碟寫入節流

`SegmentDownloader` 不是每讀到一個 chunk 就直接同步寫入磁碟，而是：

- 先在記憶體累積 buffer
- 達到 `_FLUSH_SIZE` 再 flush
- 用全域 semaphore 限制同時磁碟寫入數量

這是很務實的設計，目的是避免：

- 大量 segment 同時寫入導致 SSD 100% usage
- GUI / event loop 因 I/O 被拖慢

### 10.3 Progress / ETA / Speed

速度計算使用 rolling window `SpeedCalculator`，GUI 再依此顯示：

- 速度
- ETA
- 百分比

更新本身也有節流，避免過度轟炸 Qt 事件迴圈。

## 11. Provider 行為矩陣

| Provider | 來源 | 主要用途 | 特點 |
| --- | --- | --- | --- |
| `PixeldrainProvider` | `pixeldrain.com` | 單檔與 list | 支援 API key，list 可展開為多檔 |
| `MegaProvider` | `mega.nz` / `mega.co.nz` | MEGA 檔案標記 | 真正下載主要由 `utils/mega_api.py` 處理 |
| `GoFileProvider` | `gofile.io` | 檔案或資料夾內容 | 可用 token，沒 token 時可建 guest；資料夾預設偏向挑出主目標檔案 |
| `EroScriptsProvider` | `discuss.eroscripts.com` | 受保護附件與 short-url | 強依賴 cookies；會同步旋轉後 cookies |
| `IwaraProvider` | `iwara.tv` | Iwara 視訊 | 先用 yt-dlp 抽取，必要時走 HLS |
| `YtdlpGenericProvider` | 多數影片站 | 泛用影片解析 | 支援很多站，必要時可交由 yt-dlp 實際下載 |
| `DirectHTTPProvider` | 任意 HTTP/HTTPS | 後備方案 | 直接 HEAD + 一般檔案下載 |

### 11.1 GoFile 的重要細節

這個專案裡，GoFile 有一個值得注意的行為差異：

- 在前端 UI 側，content script 會把 GoFile 視為可能的 bundle，讓使用者先 probe 看裡面有哪些檔案。
- 但在 `QueueManager._is_bundle_url()` 裡，bundle 自動展開只明確支援 Pixeldrain list 與 MEGA folder。
- 因此 GoFile 較像是「由前端先拆選，再把實際子檔 URL 送給後端」，而不是後端統一用 bundle placeholder 解開。

這一點對閱讀程式的人非常重要，因為如果只看 UI 會以為 GoFile 跟 Pixeldrain/MEGA 完全同型，但實際後端流程不一樣。

### 11.2 EroScripts 的重要細節

EroScripts 的難點不在下載，而在登入狀態與 cookie 維持：

- short-url 需要 cookies
- Discourse 會旋轉 `_t` 等 cookie
- 若不把新 cookie 同步回設定，瀏覽器 session 可能會失效

這也是為什麼專案在 GUI 內建瀏覽器裡做了：

- CDP cookie 抽取
- cookie restore
- cookie auto-save
- auto re-login

## 12. GUI 與內建瀏覽器

### 12.1 Main Window

`funpairdl/gui/main_window.py` 提供：

- `Browser` 分頁
- `Downloads` 分頁
- 工具列
- 任務樹狀列表
- 狀態列
- 系統匣整合

主要功能：

- 新增 pair
- 暫停全部
- 恢復全部
- 清除完成項目
- 重啟 pump
- 設定視窗
- 右鍵操作單一 pair

### 12.2 下載樹狀 UI

列表是「pair 作為父節點、item 作為子節點」的結構。

顯示欄位包含：

- Name
- Status
- Progress
- Size
- Speed
- ETA

更新策略不是每次事件都直接重畫，而是：

- item/pair 先標記 dirty
- 定時批次 flush

這是為了避免高頻進度更新讓 GUI 卡頓。

### 12.3 內建瀏覽器

`funpairdl/gui/browser_widget.py` 是整個專案最複雜的 GUI 元件之一。

它負責：

- 多分頁瀏覽
- session restore
- scroll position restore
- QWebChannel 橋接
- 注入 `extension/content.js`
- CDP cookie 擷取
- EroScripts cookie restore
- MEGA SID 自動刷新

### 12.4 為什麼要用 QWebChannel

嵌入瀏覽器裡的 JS 不能直接調 Python 函式，所以用 `QWebChannel` 做橋接。

JS 可以傳送的訊息類型包含：

- `check-status`
- `send-pair`
- `resolve-url`
- `probe-url`
- `get-config`
- `send-link`
- `get-ero-credentials`
- `storage-get`
- `storage-set`

這讓內建瀏覽器可以模擬瀏覽器擴充套件的背景頁行為。

### 12.5 Cookie / Session 維護

這部分非常關鍵：

- PySide6 的 cookie signal 在某些版本不可靠
- 專案改用 Chrome DevTools Protocol 讀取所有 cookies
- 會把 EroScripts cookie 保存到 `config.json`
- 啟動時若 QWebEngine 內沒有 cookies，會從備份 restore 回去

結論是：  
內建瀏覽器不是單純的網頁容器，而是整個登入狀態維持系統的一部分。

## 13. 瀏覽器擴充套件

### 13.1 擴充套件角色

`extension/` 的擴充套件有三個主要用途：

1. 在 EroScripts 貼文頁面解析影片與 script
2. 提供右鍵選單，把單一連結送給本機下載器
3. 顯示本機後端是否在線

### 13.2 content script 做了什麼

`content.js` 是前端邏輯核心，功能非常多：

- 解析 OP 與留言區
- 解析 heading section，支援 collection 模式
- 判斷多影片貼文
- 抽取 `.funscript` 連結
- 嘗試從 DOM 偵測 script 作者
- 對影片來源排序
- 判斷 bundle link
- probe 檔案大小與內容
- 允許使用者勾選要送出的項目
- 支援畫質選擇
- 支援 auto rename
- 支援處理 lazy-loaded / cloaked posts
- 在 Discourse SPA 導頁時維持 UI
- 監控登出狀態並自動重登入

這已經不是單純「加一顆按鈕」的 content script，而是一個頁面內 mini-application。

### 13.3 background.js 做了什麼

`background.js` 主要負責：

- 建立右鍵選單
- 接收 content script 訊息
- 呼叫本機 API
- 透過 `chrome.cookies` 取出 EroScripts cookies

### 13.4 popup

`popup.html` / `popup.js` 是簡單狀態介面，用來顯示：

- 本機後端是否上線
- 版本號
- 當前 queue 數量

## 14. 本機 API 設計

API 定義在：

- `funpairdl/api/server.py`
- `funpairdl/api/routes.py`
- `funpairdl/api/schemas.py`

特點：

- 預設綁定 `127.0.0.1:9172`
- CORS 全開，方便本機 extension / 嵌入瀏覽器呼叫
- 關閉 Swagger / ReDoc

### 14.1 主要 API 一覽

| Method | Path | 用途 |
| --- | --- | --- |
| `GET` | `/api/status` | 查詢服務狀態與版本 |
| `GET` | `/api/config` | 回傳 extension 需要的設定 |
| `POST` | `/api/pair` | 新增一組 video + script 任務 |
| `POST` | `/api/resolve` | 解析短網址或需 cookies 的 URL |
| `POST` | `/api/probe` | 探測檔案資訊、大小、格式 |
| `POST` | `/api/link` | 送出單一連結 |
| `GET` | `/api/queue` | 取得佇列摘要 |
| `POST` | `/api/pair/{pair_id}/pause` | 暫停 pair |
| `POST` | `/api/pair/{pair_id}/resume` | 恢復 pair |
| `DELETE` | `/api/pair/{pair_id}` | 移除 pair |
| `POST` | `/api/pair/{pair_id}/move` | 調整 pair 順序 |
| `POST` | `/api/pair/{pair_id}/organize` | 手動整理命名 |
| `POST` | `/api/pair/{pair_id}/undo-organize` | 還原整理命名 |
| `POST` | `/api/pump/restart` | 重啟下載 pump |

### 14.2 `/api/pair` 的語意

`/api/pair` 不只是塞幾個 URL 而已，還支援：

- `preferred_resolution`
- `script_authors`
- `auto_rename`
- `eroscripts_cookies`

因此它已經不是純粹 transport 層，而是承載了前端整理後的語意資訊。

## 16. 輸出檔案整理規則

這是這個專案很有辨識度的一塊。

### 16.1 基本規則

完成後如果 `auto_rename=True`，會執行 `_organize_output()`：

- 影片改名為 `Pair.name + 原副檔名`
- script 也改成相同 base name

例如：

- `original_video.mp4` -> `My Video.mp4`
- `something.pitch.funscript` -> `My Video.pitch.funscript`

### 16.2 軸向 suffix

系統內建一套 erodeck 軸向映射，例如：

- `stroke` / `L0`
- `surge` / `L1`
- `sway` / `L2`
- `suck` / `L3`
- `twist` / `R0`
- `roll` / `R1`
- `pitch` / `R2`
- `vibe` / `V0`
- `pump` / `V1`
- `valve` / `V2`

已知軸向會保留為檔名 suffix。

### 16.3 Variant / Alt 結構

當出現以下情況時，會建立 `.alt`、`.alt1` 等資料夾：

1. 同一 canonical axis 有多個 script，發生 collision
2. `script_variant_mode = "subfolder"` 且有多位作者

例如：

- `Movie.funscript`
- `Movie.alt/Movie.alt.funscript`
- `Movie.alt1/Movie.alt1.funscript`

### 16.4 Hardlink

alt 資料夾內的影片不是完整複製，而是嘗試用 hardlink：

- 節省空間
- 保持每個 alt 資料夾看起來像完整作品目錄

並且會寫出 `.linkinfo` 記錄 hardlink 關係。

### 16.5 Undo

`undo-organize` 會：

- 刪除 `.linkinfo`
- 移除 alt 資料夾中的 hardlinked video
- 把 script 搬回根目錄
- 還原原始檔名

## 18. 已知限制與風險

### 18.1 `config.json` 會保存敏感資訊

這是目前最需要注意的地方。

`config.json` 會明文保存：

- 帳號
- 密碼
- API token
- session cookie
- MEGA SID

因此：

- 不應直接提交到公開版本控制
- 不應直接丟給外部 LLM
- 分享專案前應先清理或替換成假資料

### 18.2 Server 模式邏輯上可不需要 GUI，但 `requirements.txt` 仍包含 PySide6

也就是說：

- 程式碼結構有 headless server mode
- 但目前安裝依賴仍偏向整包桌面版

如果未來要做純後端部署，依賴拆分還需要再整理。

### 18.3 API 沒有內建 Swagger 文件

`docs_url=None`、`redoc_url=None`，所以：

- 對使用者而言比較乾淨
- 對外部整合者而言，得看原始碼或這份文件

### 18.4 有些檔案屬於歷史或輔助性質

例如：

- `mega_provider.py` 內含 `mega.py` 路徑，但實際主下載路徑已偏向 `utils/mega_api.py`
- `analyze_html.py` 與 `analysis_output.txt` 屬於 parser 設計輔助資料

閱讀專案時不要把這些誤判為唯一主路徑。

## 19. 給 LLM 的快速分析指南

如果你是要把這個專案交給另一個 LLM 分析，建議它依照下面順序讀：

1. `funpairdl/core/queue_manager.py`
2. `funpairdl/core/download_task.py`
3. `funpairdl/core/segment.py`
4. `funpairdl/providers/registry.py`
5. `funpairdl/providers/*.py`
6. `funpairdl/api/routes.py`
7. `funpairdl/gui/browser_widget.py`
8. `extension/content.js`
9. `funpairdl/persistence/settings.py`
10. `funpairdl/core/pair.py`

### 19.1 LLM 應優先掌握的不變條件

- 真正的任務單位是 `Pair`，不是單一 URL。
- `Pair` 的最終狀態由其 `items` 的狀態決定。
- provider 只負責 resolve，不一定負責最終下載。
- `ResolvedFile.is_mega` 與 `ResolvedFile.is_hls` 會改變下載路徑。
- `filename` 不是固定不變，resolve 與 organize 都可能修改。
- `queue.json` 是重要狀態來源，但啟動時會做修正與恢復。
- 內建瀏覽器不是附屬 UI，而是 cookie / session 管理核心。
- `extension/content.js` 與 GUI 內嵌瀏覽器共享頁面解析邏輯。

### 19.2 LLM 分析時應避免直接餵入的檔案

- `config.json`
- `funpairdl.log`
- `queue.json`
- `tests/__pycache__/`
- `funpairdl/**/__pycache__/`

原因：

- 有敏感資訊
- 體積大但訊號低
- 容易讓模型把 runtime 垃圾資料誤認為業務邏輯

### 19.3 如果要 debug，應優先問的問題

- 問題發生在 resolve 還是 download？
- 是單檔 URL、bundle URL、還是需要 cookies 的 URL？
- provider 回傳的是 direct、MEGA、還是 HLS？
- 問題是 pair 卡住、item 卡住、還是 segment 卡住？
- queue reload 後是否出現 stale state？
- organize_output 是否改寫了你以為固定的檔名？

## 20. 人類維護者的閱讀建議

如果你是第一次接手這個專案，建議先建立這個心智模型：

### 第一層：把它當成本機下載平台

不是單純桌面視窗，而是：

- GUI
- API
- provider 層
- queue engine
- extension / browser integration

### 第二層：把它當成配對與整理系統

這個專案最大的價值不只是下載成功，而是：

- 把混亂來源整理成 pair
- 保留作者與軸向資訊
- 重建資料夾結構

### 第三層：把它當成 session-sensitive downloader

很多問題都不是單純 HTTP 404，而是：

- cookies 過期
- short-url 需要 auth
- Discourse token 旋轉
- MEGA SID 失效

所以 session 維護是核心，不是附加功能。

## 21. 總結

`FunPairDL` 是一個以「影片 + funscript 配對下載」為核心的本機下載管理系統，具備：

- GUI 介面
- 純後端模式
- 本機 API
- 嵌入式瀏覽器
- 瀏覽器擴充套件
- 多 provider 解析
- 分段下載與重試
- 下載後命名與變體整理

它最有價值的地方不是某一個單獨模組，而是把「頁面解析、登入狀態、來源解析、背景下載、輸出整理」整合成一條完整流程。

如果你是使用者，這代表它能幫你把分散在不同來源、不同格式、不同命名風格的資源整理成可用結果。  
如果你是維護者或 LLM，這代表你必須把它當成一個多層系統來理解，而不是只看某個 `download()` 函式。
